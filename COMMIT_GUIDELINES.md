# OpenClaw Trader - Git提交规范 v1.0

## 1. 提交粒度要求
- 单次提交只做一件事，禁止一次提交包含多个不相关的修改
- 小步提交，每个功能点完成就提交，不要攒一大堆再提交
- 提交必须可以正常编译，不能提交无法编译的代码

## 2. Commit Message 格式
```
<类型>(<可选范围>): <主题>

[可选：正文]

[可选：页脚]
```

### 2.1 类型说明
| 类型 | 说明 |
|------|------|
| feat | 新增功能 |
| fix | 修复bug |
| docs | 文档修改 |
| style | 代码格式调整（空格、格式化、缺少分号等，不影响代码运行） |
| refactor | 代码重构（既不新增功能，也不修复bug的代码变动） |
| perf | 性能优化 |
| test | 测试相关修改（新增测试用例、修改测试代码等） |
| chore | 构建过程、辅助工具、依赖库修改等 |
| revert | 回滚之前的提交 |

### 2.2 范围说明
- 可选，说明本次提交影响的模块
- 例如：`feat(Portfolio): 新增持仓导入功能`
- 影响多个模块可以用`*`或者省略

### 2.3 主题要求
- 不超过50个字符
- 中文简洁明了，说明本次提交做了什么
- 结尾不用句号

### 2.4 正文要求
- 可选，用于详细说明本次提交的背景、原因、实现思路
- 每行不超过72个字符

### 2.5 页脚要求
- 可选，用于关联Issue、关闭Issue等
- 例如：`Closes #123` 表示提交后自动关闭#123号Issue

## 3. 提交示例
### 示例1：新增功能
```
feat(Trading): 新增交易风格分析功能

- 实现交易频率统计
- 实现持仓周期分析
- 实现交易风格识别算法

Closes #45
```

### 示例2：修复bug
```
fix: 修复TradingAnalysis.swift枚举语法错误

case和中文枚举值之间缺少空格导致编译失败
```

### 示例3：文档修改
```
docs: 添加代码开发规范和提交规范文档
```

### 示例4：重构
```
refactor(Network): 重构APIClient模块

- 统一错误处理逻辑
- 增加请求重试机制
- 优化缓存策略
```

## 4. 禁止行为
- ❌ 禁止提交无法编译的代码
- ❌ 禁止提交注释掉的代码
- ❌ 禁止无意义的提交信息（比如`update`、`fix`、`修改`等）
- ❌ 禁止提交超大文件（图片、资源超过10MB需要单独说明）
- ❌ 禁止在master分支直接提交，所有代码必须通过PR合并
